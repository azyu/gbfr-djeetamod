# Djeeta MOD 외부 장비 리더 프로브 설계

작성일: 2026-07-19  
대상: Granblue Fantasy: Relink Endless Ragnarok 2.0.2, Windows x64

## 1. 목표

현재 DLL 훅이 보내는 로컬 캐릭터 장비 스냅샷을 정답값으로 유지하면서, Tauri 백엔드의 외부 읽기 전용 프로브가 같은 장비 데이터를 독립적으로 찾아 읽을 수 있는지 검증한다.

첫 마일스톤은 데이터 수집 방식의 실증이다. 기존 장비 분석 화면과 대미지 미터의 동작은 바꾸지 않으며, 외부 프로브 결과는 개발 로그에만 기록한다. 이 단계에서 훅이나 `dll-syringe`를 제거하지 않는다.

## 2. 범위

### 포함

- 개발 빌드에서 `DJEETA_EXTERNAL_READER_PROBE=1`일 때만 외부 프로브 실행
- 게임 PID, 모듈 베이스와 실행 파일 SHA-256 식별
- `PROCESS_QUERY_INFORMATION | PROCESS_VM_READ` 범위의 읽기 전용 프로세스 접근
- 게임 모듈을 복사한 바이트에서 시그니처 검색
- 플레이어 전역 루트에서 현재 로컬 캐릭터 장비 스냅샷까지의 독립적인 포인터 경로
- 기존 훅과 외부 프로브가 공유하는 장비 스냅샷 디코더
- 캐릭터 키별 훅 스냅샷과 외부 스냅샷 비교
- 일치, 불일치, 비교 보류와 탐색 실패에 대한 개발 로그
- 고정 메모리 이미지를 사용하는 자동 테스트
- 실제 게임을 세 번 이상 완전히 재실행하는 수동 검증

### 제외

- 사용자 UI에 외부 프로브 상태 또는 비교 결과 표시
- 외부 프로브 결과를 장비 분석의 기본 데이터 소스로 사용
- 대미지, DoT, 스킬별 이벤트의 외부 읽기 전환
- 전투 시작·종료 판정의 외부 읽기 전환
- 게임 메모리 쓰기, 코드 패치, 원격 스레드 또는 추가 DLL 주입
- 릴리스 패키지에서 외부 프로브 실행
- 이 마일스톤에서 기존 훅 또는 `dll-syringe` 제거

## 3. 접근 방식

외부 프로브는 훅에서 레코드 또는 스냅샷 주소를 전달받지 않는다. 게임 모듈 베이스와 검증된 시그니처에서 시작해 플레이어 관리 구조를 찾고, 포인터 경로를 따라 장비 배열에 도달해야 한다.

훅은 기존 `LocalEquipmentSnapshot`을 그대로 발행한다. 백엔드는 이 이벤트를 기존 장비 분석기에 전달하는 동시에 비교기의 정답 스냅샷으로 보관한다. 외부 프로브는 독립적으로 읽은 결과를 같은 캐릭터 키의 훅 스냅샷과 비교한다.

초기 조사에서 안정적인 전역 루트를 찾기 위해 제한적인 후보 검색을 사용할 수 있다. 후보 검색은 개발 조사 수단일 뿐 최종 폴링 경로가 아니며, 절대 주소를 저장하거나 훅이 제공한 동적 주소에 의존해서는 안 된다.

## 4. 구성 요소

### 4.1 공용 장비 디코더

현재 `src-hook`에 있는 장비 배열 디코딩 규칙을 게임 프로세스 접근과 분리된 공용 Rust 모듈로 이동한다. 훅과 Tauri 백엔드는 동일한 함수를 사용해 다음 구조를 해석한다.

- 진 슬롯 수: 12
- 슬롯 stride: `0x24`
- 전체 배열 크기: `0x1B0`
- `+0x00`: 주 특성 ID
- `+0x04`: 주 특성 레벨
- `+0x08`: 보조 특성 ID
- `+0x0C`: 보조 특성 레벨
- `+0x10`: 진 ID
- `+0x14`: 장착 캐릭터 키

기존 빈 해시, 캐릭터 키 일치, 특성 레벨 범위 검증을 유지한다. 디코더는 바이트 슬라이스와 캐릭터 키만 입력받으며 프로세스 핸들, 포인터 또는 로깅에 의존하지 않는다.

### 4.2 원격 메모리 리더

Tauri 백엔드에 Windows 전용 원격 읽기 계층을 둔다. 이 계층은 프로세스 검색, 읽기 전용 핸들 관리, 모듈 정보 조회, 정확한 길이의 바이트 읽기와 포인터 읽기를 담당한다.

모든 읽기는 요청한 바이트 수 전체가 복사된 경우에만 성공한다. null, 비정렬 또는 사용자 주소 범위를 벗어난 포인터와 접근할 수 없는 메모리는 오류로 처리한다. 오류가 기존 파서나 장비 분석 상태로 전파되어서는 안 된다.

### 4.3 시그니처와 포인터 경로 해석기

프로브는 원격 게임 모듈을 로컬 버퍼로 복사한 뒤 해당 버퍼에서 시그니처를 검색한다. 검색 결과는 정확히 한 개여야 한다. 0개 또는 여러 개이면 해당 게임 빌드를 지원하지 않는 것으로 기록하고 폴링을 시작하지 않는다.

시그니처 결과는 모듈 상대 주소로 해석한다. RIP-relative 명령 피연산자를 해석해 플레이어 전역 루트를 얻고, 검증된 오프셋 경로로 현재 로컬 플레이어 레코드와 장비 스냅샷에 도달한다. ASLR에 의해 바뀌는 절대 주소는 구성이나 로그의 재사용 가능한 입력으로 취급하지 않는다.

### 4.4 외부 프로브 실행기

외부 프로브는 다음 두 조건을 모두 만족할 때만 시작한다.

1. Rust 디버그 빌드이다.
2. 환경 변수 `DJEETA_EXTERNAL_READER_PROBE`의 값이 정확히 `1`이다.

게임이 종료되면 핸들을 닫고 탐색 상태를 폐기한다. 게임을 다시 찾으면 모듈과 시그니처부터 새로 확인한다. 릴리스 빌드에서는 환경 변수가 설정돼도 프로브를 시작하지 않는다.

### 4.5 스냅샷 비교기

비교기는 훅과 외부 프로브의 최신 스냅샷을 캐릭터 키로 연결한다. 외부 프로브는 250ms마다 후보 스냅샷을 확인한다. 장비 변경 도중의 찢어진 읽기를 피하기 위해 외부 배열을 50ms 간격으로 두 번 읽으며, 두 바이트 배열이 동일할 때만 디코딩하고 비교한다.

비교 결과는 다음 중 하나이다.

- `MATCH`: 캐릭터 키와 모든 장비 특성 소스가 일치
- `MISMATCH`: 같은 캐릭터의 안정된 두 결과가 다름
- `DEFERRED`: 외부 두 읽기가 다르거나 한쪽 스냅샷이 아직 없음
- `UNAVAILABLE`: 시그니처, 포인터 또는 메모리 읽기 검증 실패

## 5. 데이터 흐름

```text
기존 DLL 훅
  -> LocalEquipmentSnapshot
     -> 기존 EquipmentStatus와 React 갱신
     -> 개발용 비교기의 훅 정답 캐시

외부 프로브
  -> 게임 PID와 모듈 식별
  -> 모듈 바이트 시그니처 검색
  -> 플레이어 루트와 포인터 경로 해석
  -> 장비 배열 1차 읽기
  -> 짧은 간격 후 장비 배열 2차 읽기
  -> 동일할 때 공용 디코더 적용
  -> 캐릭터 키별 훅 정답과 비교
  -> 개발 로그 기록
```

외부 프로브가 실패하거나 중단돼도 기존 훅 파이프와 장비 분석 데이터 흐름은 계속 동작해야 한다.

## 6. 로그 정책

프로브 시작 시 다음 정보를 한 번 기록한다.

- 게임 PID
- 게임 실행 파일 SHA-256
- 모듈 베이스
- 시그니처 일치 개수와 선택된 모듈 상대 주소
- 프로브가 요청한 Windows 프로세스 권한

비교 성공 시 캐릭터 키, 스냅샷 해시, 유효한 장비 슬롯 수와 `MATCH`를 기록한다. 불일치 시 슬롯별로 다음 필드의 차이만 기록한다.

- 진 ID
- 주 특성 ID와 레벨
- 보조 특성 ID와 레벨
- 장착 캐릭터 키

원본 `0x1B0` 바이트 전체, 플레이어 표시 이름이나 다른 개인 식별 정보는 로그에 남기지 않는다. 같은 결과의 반복 로그는 억제한다.

## 7. 오류 처리

- 게임 프로세스를 찾지 못하면 기존 검색 주기를 방해하지 않고 대기한다.
- 실행 파일 해시가 검증 대상과 다르면 해시와 미지원 상태를 기록하고 읽기를 중단한다.
- 시그니처가 정확히 한 번 일치하지 않으면 포인터를 역참조하지 않는다.
- 부분 메모리 읽기는 성공으로 간주하지 않는다.
- 두 외부 배열이 다르면 장비 변경 중인 것으로 간주하고 비교를 보류한다.
- 디코딩 검증 실패는 빈 장비나 0레벨로 변환하지 않는다.
- 프로브 오류는 `HookStatus`, 미터 연결 상태 또는 `EquipmentStatus.connected`를 변경하지 않는다.
- 동일한 반복 실패 로그는 5초에 한 번만 기록한다.

## 8. 테스트 전략

### 8.1 단위 테스트

실제 Windows 프로세스 대신 주소와 바이트를 제공하는 메모리 읽기 인터페이스를 사용한다.

- 시그니처가 정확히 한 번 일치할 때만 루트를 선택
- 시그니처 0개와 복수 일치 거부
- 서로 다른 모듈 베이스에서도 같은 상대 주소와 포인터 경로 해석
- 정상 포인터 체인 읽기
- null, 범위 밖 포인터와 부분 읽기 거부
- 동일한 외부 이중 읽기는 안정된 스냅샷으로 승인
- 서로 다른 외부 이중 읽기는 `DEFERRED`
- 같은 캐릭터의 동일 스냅샷은 `MATCH`
- 슬롯별 차이가 있는 스냅샷은 `MISMATCH`와 제한된 차이 정보 생성
- 같은 비교 결과의 중복 로그 억제
- 환경 변수가 없거나 `1`이 아니면 프로브 비활성
- 릴리스 빌드에서는 환경 변수가 `1`이어도 프로브 비활성

모든 새 동작은 실패하는 테스트를 먼저 확인한 뒤 최소 구현으로 통과시킨다.

### 8.2 회귀 검증

프로젝트 필수 검증 명령을 모두 실행한다.

```powershell
npm ci
npm run format-check
npm run lint
npm run tsc
npm test -- --run
npm run build
cargo build --release --locked --package hook
cargo test --workspace --all-targets --locked
npm run tauri build -- --bundles nsis
```

이번 단계는 장비 수집 방식의 개발용 실증이므로 패키지 해시 기록은 실제 배포 패키지를 교체하기로 결정한 경우에만 갱신한다.

### 8.3 실제 게임 검증

`DJEETA_EXTERNAL_READER_PROBE=1`을 설정한 디버그 빌드에서 다음을 수행한다.

1. 게임과 MOD를 완전히 종료한 상태에서 시작한다.
2. 서로 다른 캐릭터 두 명 이상을 관찰한다.
3. 각 캐릭터에서 진 하나를 교체하고 원래대로 복구한다.
4. 안정된 각 상태에서 외부 결과가 훅 결과와 `MATCH`인지 확인한다.
5. 게임 프로세스를 완전히 종료하고 총 세 번 이상 다시 실행한다.
6. 매 실행에서 절대 주소는 달라도 같은 시그니처와 포인터 경로가 성립하는지 확인한다.
7. 프로브를 비활성화한 평상시 개발 실행과 릴리스 빌드가 기존과 동일하게 동작하는지 확인한다.

## 9. 성공 기준

- 외부 프로브는 훅이 제공한 동적 주소 없이 장비 배열을 찾는다.
- 외부 프로브는 읽기와 조회 권한만 사용하며 게임 메모리를 변경하지 않는다.
- 게임을 세 번 이상 완전히 재실행해도 지원 대상 2.0.2 실행 파일에서 경로가 다시 해석된다.
- 서로 다른 캐릭터와 진 교체·복구 상태의 안정된 스냅샷이 모두 훅 결과와 일치한다.
- 프로브 실패가 기존 대미지 미터, 전투 기록, 장비 분석 UI와 연결 상태에 영향을 주지 않는다.
- 환경 변수로 활성화하지 않은 개발 실행과 모든 릴리스 실행에서 프로브가 동작하지 않는다.
- 자동 테스트와 프로젝트 필수 빌드·린트·테스트가 모두 통과한다.

이 성공 기준을 만족한 후에만 장비 분석의 기본 데이터 소스를 외부 리더로 전환하는 별도 설계를 시작한다.
