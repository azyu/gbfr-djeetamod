# GitHub Release Auto-Update Design

**Date:** 2026-07-22

## Goal

Add a signed, user-approved Windows auto-update path for Djeeta MOD using the
public `azyu/gbfr-djeetamod` GitHub repository. Preserve the existing canonical
NSIS packaging checks, game-process safety rules, repeat-quest restoration, and
per-user application data.

## Repository and Release Model

The current source repository, GitHub Actions workflows, and GitHub Releases
live together at `azyu/gbfr-djeetamod`. The local `origin` remote points to this
repository over SSH, while the existing `awa` remote remains an upstream
reference.

Stable update metadata is downloaded from:

`https://github.com/azyu/gbfr-djeetamod/releases/latest/download/latest.json`

GitHub's stable `releases/latest` route excludes drafts and prereleases. A
release is therefore invisible to production clients until every artifact has
been uploaded and the draft is deliberately published.

Release publication is initiated with a manual GitHub Actions
`workflow_dispatch`, not directly by pushing a tag. The workflow:

1. checks that `package.json`, `src-tauri/Cargo.toml`, and
   `src-tauri/tauri.conf.json` contain one matching semantic version;
2. requires the requested release version to match that project version and
   refuses an existing tag or release;
3. runs the existing `npm.cmd run package:nsis` command as the single canonical
   Windows packaging entry point;
4. signs and locates the updater bundle and signature generated by Tauri;
5. generates a `latest.json` manifest for `windows-x86_64` using the immutable
   versioned release-asset URL and the exact contents of the signature file;
6. commits the README and smoke-test hash updates produced by the canonical
   packaging script together, then tags that documentation-only release commit;
7. creates a draft release and uploads the NSIS installer, updater archive,
   updater signature, and `latest.json`;
8. downloads or queries the uploaded assets to verify names, hashes, manifest
   URLs, and signature contents; and
9. publishes the draft only after every verification succeeds.

The workflow receives only `contents: write` permission. It uses the
repository-scoped `GITHUB_TOKEN`; no cross-repository personal access token is
needed. Third-party actions are pinned to full commit SHAs.

## Updater Configuration and Signing

The application continues to use Tauri 1 and its built-in updater. The Tauri
dependency enables its updater feature, and `src-tauri/tauri.conf.json` changes
to:

- activate the updater;
- use the Djeeta MOD GitHub `latest.json` endpoint;
- disable Tauri's built-in dialog so the existing React UI owns the Korean user
  experience;
- use the recommended passive Windows installer presentation; and
- replace the inherited upstream public key with a newly generated Djeeta MOD
  updater public key.

The matching private key and its password are never committed. They are stored
as protected GitHub Actions secrets and backed up separately in a secure
offline location. Release builds receive them through Tauri's documented
`TAURI_PRIVATE_KEY` and `TAURI_KEY_PASSWORD` environment variables. Losing the
private key would prevent installed clients from accepting later updates.

Updater signing authenticates the downloaded update package but is separate
from Windows Authenticode signing. This feature does not claim to make the
currently unsigned executable trusted by SmartScreen or antivirus products.

## Application Experience

The application checks once, without blocking startup, after the management UI
is ready. It does not poll repeatedly in the background. The Settings page also
provides an explicit **업데이트 확인** action.

When no update exists, an automatic check remains silent and a manual check
reports that the current version is latest. Network failures, GitHub outages,
invalid metadata, and signature failures never stop the damage meter or other
application features. Automatic-check failures are logged without an intrusive
dialog; manual-check failures produce a concise Korean error message.

When a newer semantic version exists, the UI shows the installed version, new
version, release notes when present, and explicit **나중에** and **업데이트**
actions. The app never downloads or installs an update solely because startup
completed. Only a user confirmation begins installation.

The frontend models these states explicitly: idle, checking, available,
up-to-date, preparing, installing, and error. Concurrent startup and manual
checks share one in-flight operation so duplicate dialogs or installs cannot
occur.

## Game and Repeat-Quest Safety

Installing while Granblue Fantasy: Relink is running is prohibited because the
injected and bundled hook may still be in use. Djeeta MOD never terminates the
game automatically.

Before installation, a backend preparation command performs one serialized
transaction:

1. detect whether the exact supported game process is running;
2. if repeat quest is enabled, restore both patched instructions and verify the
   original bytes by read-back;
3. refuse installation if restoration fails;
4. ask the user to close the game when it remains running; and
5. recheck immediately before allowing the updater installation to start.

The installer begins only after the game is confirmed stopped. The normal
application exit cleanup remains in place as defense in depth. Update
preparation does not delete or rewrite `logs.db`, settings, window geometry, or
other per-user data. The current-user NSIS installer upgrades the application
in place and Tauri restarts it after a successful installation.

## Bootstrap and Version Policy

Versions use semantic versioning and releases use matching `vX.Y.Z` tags.
Tauri accepts only a version newer than the installed application version.
Rollback is performed by releasing a corrected build with a higher version;
published assets for an existing version are never replaced.

An already installed build whose updater is inactive cannot acquire this
feature automatically. If the existing `0.1.1` build has been distributed,
those users must install the first updater-enabled build manually. Rebuilding a
different binary as the same `0.1.1` version does not trigger an update for an
existing `0.1.1` installation. The first end-to-end update acceptance path is
an updater-enabled `0.1.1` installation discovering and installing `0.1.2`.

## Failure Handling

- A missing or malformed `latest.json` is treated as an update-check failure,
  not an application failure.
- An invalid signature prevents installation and leaves the current version
  running.
- A partial workflow run remains a draft and cannot become the stable latest
  release.
- A game-process race is handled by checking again immediately before install.
- A repeat-quest restoration failure blocks installation and tells the user to
  disable the feature or restart the game before retrying.
- Update installation does not silently downgrade, replace assets under an
  existing tag, or bypass signature verification.

## Testing and Acceptance

Implementation follows test-first changes for update state handling, Tauri
configuration, version consistency, manifest generation, and release artifact
selection.

Automated acceptance requires:

- frontend tests for startup/manual checks, duplicate-check suppression, user
  cancellation, install confirmation, and non-fatal errors;
- backend tests for update preparation, repeat-quest restoration failures, and
  the game-running gate where these can be isolated from live process access;
- configuration tests that require the exact GitHub endpoint, new public key,
  updater activation, custom dialog mode, and passive Windows install mode;
- PowerShell tests for version agreement, updater artifact selection,
  `latest.json` generation, signature embedding, and refusal of stale or
  ambiguous artifacts;
- the repository's full frontend and Rust verification suites;
- a successful canonical NSIS release package with equal release and bundled
  `hook.dll` hashes; and
- a GitHub Actions dry run that leaves a draft until asset verification passes.

Manual acceptance uses an offline or private game session and covers:

- updater-enabled `0.1.1` detecting `0.1.2`, cancellation, installation,
  restart, and displayed-version confirmation;
- offline GitHub, missing manifest, and invalid-signature behavior;
- installation refusal while the game is running;
- repeat quest enabled before update preparation and verified restoration;
- preservation of combat logs, settings, and window geometry; and
- final installer and hook hashes in both active release documents.

Automated tests and a successful updater installation do not establish game
2.0.2 compatibility. That claim remains gated by the existing manual smoke-test
checklist.

## Non-Goals

- Tauri 2 migration
- prerelease or beta update channels
- staged percentage rollouts or update telemetry
- automatic game termination
- silent updates without user approval
- Windows Authenticode code signing
- migration of application data or `logs.db`
