# Djeeta MOD 읽기 전용 보유 진 인벤토리 프로브 설계

작성일: 2026-07-20
대상: Granblue Fantasy: Relink Endless Ragnarok 2.0.2, Windows x64
상태: 구현 전 승인된 설계

## 1. 목표

게임의 진 인벤토리 화면을 연 상태에서 사용자가 한 번 요청하면, Tauri 백엔드가 게임 프로세스의 읽기 가능한 private memory를 제한적으로 스캔해 전체 보유 진 배열 후보를 찾는다.

이번 단계는 개발용 실증이다. 후보가 정확히 하나이고 두 번의 직접 읽기가 안정적으로 일치할 때만 성공으로 기록한다. 결과는 개발 로그에만 남기며, 검증이 끝나기 전에는 제품 기능이나 2.0.2 전체 인벤토리 지원으로 표시하지 않는다.

## 2. 기존 구현에서 유지할 원칙

이번 설계는 다음 기존 커밋에서 확립한 외부 장비 리더 구조를 확장한다.

- `2f364bd`: 외부 장비 리더 설계
- `6359835`: 테스트 우선 구현 계획
- `9423e18`: 프로세스 접근과 분리된 공용 장비 디코더
- `5c39fe6`: 시그니처와 포인터 경로 해석기
- `fe14963`: 최소권한 Windows 메모리 읽기 계층
- `49c89b1`: 제한된 digest 기반 비교
- `8dbc530`: 디버그 빌드와 환경 변수의 이중 opt-in
- `c5b997d`: 파티 전체, 재연결, 반복 실행 검증

다음 원칙은 변경하지 않는다.

- 앱 manifest는 `asInvoker`를 유지한다.
- NSIS 설치는 `currentUser`를 유지한다.
- 외부 프로세스 핸들은 `PROCESS_QUERY_LIMITED_INFORMATION | PROCESS_VM_READ`만 요청한다.
- 게임 메모리 쓰기, 코드 패치, 원격 스레드, 추가 DLL 주입을 사용하지 않는다.
- 기존 데미지 훅, named pipe, 장착 진 분석 경로와 연결 상태를 변경하지 않는다.
- 게임 실행 파일 SHA-256이 검증된 2.0.2 값과 다르면 메모리를 스캔하지 않는다.

## 3. 범위

### 포함

- 디버그 빌드에서 `DJEETA_INVENTORY_PROBE=1`일 때만 활성화되는 수동 1회 캡처
- 장비 분석 화면의 개발 전용 `보유 진 캡처` 버튼
- committed, private, readable, non-guard memory 영역 열거
- `0x24` 보유 진 레코드 디코딩과 연속 배열 후보 탐색
- 후보 없음, 복수 후보, 불안정한 후보의 명시적 거부
- 실행 시간과 총 읽기 양 제한
- 원본 진 목록을 포함하지 않는 제한된 개발 로그
- 합성 메모리 자동 테스트와 실제 게임 반복 검증 체크리스트

### 제외

- 전체 보유 진 목록을 사용자 UI에 표시
- 진 검색, 필터, 정렬 또는 빌드 시뮬레이터
- 보유 진 데이터의 데이터베이스 저장
- release 빌드에서 프로브 실행
- 상대 프로젝트의 Python, Pymem, Electron, code cave 또는 에셋 도입
- 가장 큰 후보를 자동으로 인벤토리로 간주하는 휴리스틱
- 현재 DLL 훅 또는 장착 진 데이터 소스 교체

## 4. 구조

### 4.1 공용 보유 진 레코드 디코더

`equipment-core`에 프로세스 접근과 무관한 보유 진 레코드 디코더를 추가한다. 입력은 정확히 한 레코드의 바이트와 2.0.2 진·특성 ID 카탈로그이며, 출력은 다음 필드를 가진 구조체다.

- 주 특성 ID와 레벨: `+0x00`, `+0x04`
- 보조 특성 ID와 레벨: `+0x08`, `+0x0C`
- 진 ID: `+0x10`
- 장착 캐릭터 키 후보: `+0x14`
- 진 레벨: `+0x18`
- 획득 순번: `+0x1C`
- 알림/상태 원본 값: `+0x20`

`+0x14`는 preset 번호로 추론하지 않는다. 현재 검증된 장착 12칸 구조에서는 캐릭터 키이므로, 전체 인벤토리에서도 통제 표본으로 의미가 확인될 때까지 원본 `u32` 값과 `EMPTY_HASH` 여부만 유지한다.

레코드 디코더는 다음을 거부한다.

- `0x24`바이트보다 짧은 입력
- 2.0.2 카탈로그에 없고 빈 값도 아닌 진 ID
- 2.0.2 카탈로그에 없고 빈 값도 아닌 특성 ID
- `0..=30` 범위 밖의 진 또는 특성 레벨
- 빈 진인데 진·특성 레벨이나 특성 ID가 남아 있는 모순된 레코드

빈 값은 `0`과 기존 `EMPTY_HASH`를 허용한다. 새로운 ID를 추정하거나 임의 이름으로 정상화하지 않는다.

### 4.2 Windows 메모리 영역 열거

기존 `RemoteProcess`에 `VirtualQueryEx` 기반 영역 열거를 추가한다. 반환되는 영역은 base address, byte length와 protection metadata만 포함한다.

스캐너는 다음 조건을 모두 만족하는 영역만 읽는다.

- `MEM_COMMIT`
- `MEM_PRIVATE`
- 읽기 가능한 protection
- `PAGE_GUARD`와 `PAGE_NOACCESS`가 아님

읽기는 4MiB chunk로 수행하고 `0x24 - 1`바이트 이상을 겹쳐 chunk 경계의 레코드를 놓치지 않는다. 부분 읽기는 성공으로 취급하지 않는다.

### 4.3 인벤토리 후보 스캐너

스캐너는 4바이트 정렬 위치에서 레코드 디코더를 시도하고, stride `0x24`로 이어지는 유효 레코드를 하나의 run으로 묶는다.

성공 후보는 다음 조건을 모두 만족해야 한다.

- 13개 이상의 연속 레코드
- 6개 이상의 비어 있지 않은 진
- 모든 비어 있지 않은 레코드가 공용 디코더를 통과
- 인접 chunk가 같은 run을 중복 보고하지 않음

13개 미만의 run은 장착 12칸 스냅샷과 구분할 수 없으므로 전체 인벤토리 후보로 승인하지 않는다. 이 제한 때문에 보유 진이 적은 초기 저장 데이터는 이번 개발 프로브의 지원 범위가 아니다.

모든 영역을 스캔한 뒤 조건을 만족하는 run이 정확히 하나일 때만 안정성 확인으로 진행한다. 0개이면 `UNAVAILABLE`, 둘 이상이면 `AMBIGUOUS`다. 후보 크기로 임의 선택하지 않는다.

### 4.4 안정성 확인

유일 후보의 base address와 run length를 사용해 원격 메모리를 다시 직접 읽는다. 첫 읽기 후 50ms를 기다려 같은 범위를 다시 읽는다.

- 두 읽기의 길이와 바이트가 같고 모든 레코드가 다시 디코딩되면 `STABLE`
- 주소 범위를 다시 읽을 수 없으면 `UNAVAILABLE`
- 두 바이트 배열이 다르면 `UNSTABLE`

성공 결과에는 전체 run 길이, 비어 있지 않은 진 수, 장착 캐릭터 키 후보의 고유 개수와 snapshot digest만 포함한다.

### 4.5 실행기와 동시성

Tauri backend는 다음 두 명령을 제공한다.

- `inventory_probe_available`: 디버그 빌드이며 환경 변수 값이 정확히 `1`일 때만 `true`
- `capture_inventory_probe`: 조건을 다시 검증한 뒤 한 번의 스캔 실행

릴리스 빌드에도 inert guard는 남을 수 있지만 `inventory_probe_available`은 항상 `false`이며 capture 명령은 메모리 접근 전에 거부된다.

스캔은 blocking worker에서 실행해 Tauri async runtime과 UI를 막지 않는다. 프로세스 내 atomic 실행 플래그로 동시 두 번째 요청을 거부하고, 성공과 실패를 포함한 모든 종료 경로에서 플래그를 해제한다.

한 번의 실행은 다음 중 먼저 도달한 한계에서 중단한다.

- 경과 시간 60초
- 요청한 총 읽기 양 16GiB

## 5. UI

장비 분석 화면은 시작 시 `inventory_probe_available`을 호출한다. 결과가 `true`일 때만 `보유 진 캡처` 버튼과 다음 안내를 표시한다.

> 게임에서 진 인벤토리 화면을 연 뒤 캡처하세요. 결과는 개발 로그에만 기록됩니다.

실행 중에는 버튼을 비활성화한다. 명령 완료 후 UI에는 진 목록이나 주소를 표시하지 않는다.

- 성공: `캡처 완료 — 개발 로그 확인`
- 실패: 게임 없음, 지원하지 않는 EXE, 후보 없음, 복수 후보, 불안정한 읽기, 실행 한도 초과 중 하나의 제한된 문구

개발용 버튼과 문구는 한국어와 영어 리소스에 모두 추가한다. 브라우저 콘솔이나 Zustand store에는 인벤토리 내용을 저장하지 않는다.

## 6. 로그 정책

로그 이벤트는 `INVENTORY_PROBE` 접두사를 사용한다.

- 시작: PID, 게임 EXE SHA-256, 요청 권한
- 진행: 영역 수, 누적 요청 바이트, 경과 시간
- 후보: base address, run length, 비어 있지 않은 진 수, digest
- 종료: `STABLE`, `UNAVAILABLE`, `AMBIGUOUS`, `UNSTABLE`, `LIMIT_EXCEEDED`

다음을 기록하지 않는다.

- 진별 ID, 특성 ID 또는 레벨의 전체 목록
- raw memory bytes
- 캐릭터 이름 또는 플레이어 표시명
- 전체 JSON inventory dump

base address는 현재 실행의 개발 진단값일 뿐 재사용 가능한 설정이나 캐시로 저장하지 않는다.

## 7. 오류 처리

- 게임을 찾지 못하면 `GAME_NOT_RUNNING`으로 종료한다.
- EXE SHA-256이 고정값과 다르면 영역 열거 전에 `UNSUPPORTED_GAME`으로 종료한다.
- `VirtualQueryEx` 또는 정확한 길이의 메모리 읽기가 실패한 영역은 원인과 주소만 제한적으로 기록하고, 후보의 일부로 사용하지 않는다.
- 시간 또는 읽기 양 한계를 넘으면 부분 결과를 성공 후보로 승격하지 않는다.
- 후보가 복수이면 주소나 크기 순으로 하나를 고르지 않는다.
- 오류는 `HookStatus`, meter connection, `EquipmentStatus` 또는 encounter parser에 전달하지 않는다.
- capture 오류 이후에도 기존 데미지 미터와 장착 진 분석은 계속 동작해야 한다.

## 8. 자동 테스트

구현은 실패하는 테스트를 먼저 추가한 뒤 최소 코드로 통과시킨다.

### 8.1 공용 디코더

- 정상 `0x24` 레코드의 모든 필드 해석
- 정확한 빈 레코드 승인
- 부분 레코드 거부
- 알 수 없는 진과 특성 ID 거부
- 범위 밖 레벨 거부
- 모순된 빈 레코드 거부

### 8.2 영역 열거와 스캐너

- committed/private/readable/non-guard 영역만 선택
- chunk 경계에 걸친 레코드 run 탐색
- 겹친 chunk가 같은 후보를 중복 보고하지 않음
- 12개 run 제외
- 13개 이상이지만 비어 있지 않은 진이 6개 미만인 run 제외
- 유일 후보 승인
- 후보 0개와 복수 후보 구분
- 두 안정성 읽기가 다르면 `UNSTABLE`
- 부분 읽기 거부
- 60초와 16GiB 한도 처리

### 8.3 실행 조건과 UI

- 디버그 빌드와 환경 변수 `1` 조합에서만 활성화
- `true`, `01`, 빈 값과 release 빌드는 비활성
- 동시 실행 거부와 종료 후 실행 플래그 복구
- 비활성일 때 버튼 미표시
- 실행 중 버튼 비활성화
- 성공과 제한된 오류 문구 렌더링
- 기존 `asInvoker`, `currentUser`, updater 비활성 설정 유지
- 소스가 `PROCESS_VM_WRITE`, `PROCESS_VM_OPERATION`, `VirtualAllocEx`, `WriteProcessMemory`를 프로브 경로에 추가하지 않음

### 8.4 회귀 검증

프로젝트의 필수 검증 명령을 모두 실행한다.


패키징 결과를 배포 후보로 채택할 때만 DLL과 installer hash를 README 및 smoke-test 문서에 갱신한다.

## 9. 실제 게임 검증

`DJEETA_INVENTORY_PROBE=1`을 설정한 디버그 빌드에서 관리자 권한 없이 다음을 수행한다.

1. 게임 2.0.2와 Djeeta MOD를 시작한다.
2. 진 인벤토리 화면을 열고 수동 캡처한다.
3. 유일한 `STABLE` 후보와 게임 UI의 보유 진 수를 대조한다.
4. 게임 화면의 정렬과 필터만 변경한 뒤 다시 캡처해 digest가 같은지 확인한다.
5. 진을 실제로 추가, 처분 또는 강화한 뒤 다시 캡처해 예상한 개수 또는 digest 변화가 있는지 확인한다.
6. 게임과 앱을 완전히 종료하고 총 세 번 이상 재실행한다.
7. 매 실행에서 절대 주소가 달라도 다시 유일한 안정 후보를 찾는지 확인한다.
8. 각 실행에서 기존 데미지 미터와 장착 진 분석이 계속 동작하는지 확인한다.
9. 환경 변수가 없는 debug 실행과 release 빌드에서 버튼이 노출되지 않고 capture가 실행되지 않는지 확인한다.

실제 결과는 `docs/testing/game-2.0.2-inventory-probe.md`에 PID, EXE hash, 후보 요약, UI 대조값과 판정을 기록한다. 원본 메모리나 전체 진 목록은 문서에 넣지 않는다.

## 10. 성공 기준

- 관리자 권한 없이 전체 인벤토리 후보를 읽는다.
- 게임 메모리를 쓰거나 추가 코드를 주입하지 않는다.
- 고정 2.0.2 EXE에서 유일한 안정 후보만 승인한다.
- 게임 UI의 보유 진 수와 프로브 결과가 일치한다.
- 정렬과 필터 변경은 digest를 바꾸지 않고 실제 인벤토리 변경은 결과에 반영된다.
- 게임을 세 번 이상 완전히 재실행해도 후보를 다시 찾는다.
- 프로브 실패와 비활성 상태가 기존 meter, parser, hook handshake와 장착 진 분석에 영향을 주지 않는다.
- 자동 테스트와 필수 검증 명령이 통과한다.
- release 빌드에서 사용자에게 노출되거나 실행되지 않는다.

이 기준을 모두 충족한 뒤에만 전체 보유 진 목록 UI를 별도 설계·계획·구현 단계로 시작한다.
