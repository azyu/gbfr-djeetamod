# 일반 아이템 보유 한도 경고 설계

## 목표

Djeeta MOD 관리 창의 사이드바에 `아이템 분석` 메뉴를 추가한다. 이 화면은 최대
보유량이 999인 모든 일반 아이템을 읽고, 현재 수량이 900 이상인 항목만 보여 주어
보상을 받기 전에 보유 한도에 가까운 아이템을 정리할 수 있게 한다.

## 범위

포함 범위는 다음과 같다.

- 최대 보유량 999가 적용되는 모든 일반 아이템
- 관리 창의 새 사이드바 메뉴와 전용 화면
- 화면 진입 시 한 번의 자동 검사
- 사용자가 누르는 `새로고침` 버튼을 통한 재검사
- 한국어 아이템 이름, 현재 수량, 최대 수량 표시
- Granblue Fantasy: Relink Endless Ragnarok 2.0.2에서 검증된 읽기 전용 접근

다음은 포함하지 않는다.

- 진, 무기, 가호, 캐릭터 장비처럼 개별 레코드로 관리되는 자산
- 임계값 설정 기능
- 백그라운드 폴링, 토스트 알림 또는 전투 중 지속 감시
- 아이템 판매, 교환, 폐기 등 게임 데이터를 변경하는 기능
- 다른 게임 버전에 대한 추정 지원

## 사용자 경험

사이드바의 기존 `진 특성 상한 분석`과 `전투 기록` 사이에 `아이템 분석` 메뉴를
추가한다. 메뉴를 선택하면 전용 경로로 이동한다.

화면은 처음 열릴 때 아이템 인벤토리를 한 번 검사한다. 검사 중에는 진행 상태를
표시하고 중복 실행을 막는다. 이후 데이터는 사용자가 `새로고침`을 눌렀을 때만
갱신한다.

성공한 결과는 다음 규칙으로 표시한다.

- 수량이 900 이상인 항목만 포함한다.
- 수량 내림차순으로 정렬하고, 수량이 같으면 번역된 아이템 이름 순으로 정렬한다.
- 각 행은 한국어 아이템 이름과 `현재 수량 / 999`를 표시한다.
- 조건을 만족하는 항목이 없으면 `보유 한도에 가까운 아이템이 없습니다`라는 빈
  상태를 표시한다.
- 이전에 성공한 결과가 있는 상태에서 새로고침이 실패하면 기존 결과를 유지하면서
  오류를 함께 표시한다.

## 아키텍처

일반 아이템 분석은 주입된 DLL 훅이 아니라 Tauri 백엔드의 외부 읽기 전용
프로브로 구현한다. 기존 장비 및 진 프로브와 같은 프로세스 검색, 실행 파일
SHA-256 고정, `PROCESS_QUERY_INFORMATION | PROCESS_VM_READ` 접근 원칙을 재사용한다.
전투 훅, 프로토콜 메시지, 미터 연결 상태에는 영향을 주지 않는다.

백엔드는 세 단위로 분리한다.

1. 일반 아이템 레코드 디코더는 프로세스 접근 없이 바이트를 아이템 ID와 수량으로
   변환하고 유효성을 검사한다.
2. 로케이터는 검증된 2.0.2 메모리 구조에서 일반 아이템 인벤토리를 찾고, 두 번
   읽은 스냅샷이 같은지 확인한다.
3. Tauri 명령은 프로브 실행을 직렬화하고 UI에 필요한 최소 결과만 반환한다.

프론트엔드는 별도의 `ItemAnalysis` 페이지와 응답 타입을 둔다. 기존 장비 분석
스토어나 이벤트를 재사용하지 않아 두 기능의 상태와 오류가 섞이지 않게 한다.

## 데이터 계약

성공 응답은 다음 정보만 반환한다.

- 검사 시각
- 임계값 `900`
- 최대 수량 `999`
- 조건을 만족한 각 항목의 아이템 ID, 번역된 이름을 찾기 위한 키, 현재 수량

프론트엔드는 번들된 아이템 번역 리소스로 이름을 표시한다. 번역이 없는 유효한
아이템은 숨기지 않고 8자리 16진수 ID를 포함한 `알 수 없는 아이템`으로 표시한다.
중복 ID가 발견되면 임의로 합산하지 않고 스냅샷을 유효하지 않은 것으로 거부한다.

## 안전성과 오류 처리

프로브는 다음 조건을 모두 만족해야 한다.

- 게임 프로세스에 읽기 전용 권한만 요청한다.
- 고정된 2.0.2 실행 파일 SHA-256이 아니면 메모리를 해석하지 않는다.
- 주소와 레코드 레이아웃은 통제된 실제 게임 검증으로 확정한 값만 사용한다.
- 동시에 한 번만 실행한다.
- 두 번의 연속 읽기가 다르면 불안정한 결과로 거부한다.
- 전체 아이템 목록, 원시 메모리, 재사용 가능한 절대 주소를 로그에 남기지 않는다.
- 실패가 `HookStatus`, 미터 연결 상태, 장비 분석 상태 또는 전투 기록에 전달되지
  않는다.

UI는 게임 미실행, 미지원 실행 파일, 데이터 위치를 찾지 못함, 스냅샷 불안정,
이미 검사 중, 내부 오류를 서로 다른 상태로 표시한다. 실패나 미검증 상태를
`900 이상인 아이템 없음`으로 표시하지 않는다.

## 단계적 활성화

메모리 구조가 아직 검증되지 않았으므로 첫 단계는 디버그 빌드와 명시적 환경 변수
뒤에서만 실행되는 조사용 프로브다. 이 단계에서는 후보 주소, 원시 레코드 또는 전체
인벤토리를 사용자 화면에 공개하거나 릴리스 기능으로 활성화하지 않는다.

다음 검증을 모두 통과한 후 같은 디코더와 로케이터를 일반 `아이템 분석` 화면에
연결한다.

1. 알려진 일반 아이템 ID와 수량이 게임 UI와 일치한다.
2. 한 아이템 수량만 변경했을 때 해당 레코드만 예상대로 변경된다.
3. 899, 900, 998, 999 경계값이 정확히 판정된다.
4. 아이템 메뉴 정렬과 필터 변경이 결과에 영향을 주지 않는다.
5. 게임 재시작 후에도 같은 논리 인벤토리를 다시 찾는다.
6. 기존 장비 분석, 미터, 전투 기록 및 반복 퀘스트 기능이 그대로 동작한다.

검증이 끝나기 전에는 일반 아이템 인벤토리 호환성을 주장하지 않는다.

## 테스트

구현은 다음 자동 검증을 포함한다.

- 일반 아이템 레코드의 정상, 빈 슬롯, 잘못된 ID, 잘못된 수량 디코딩 테스트
- 899 제외, 900 포함, 999 포함 경계값 테스트
- 중복 ID, 불안정한 두 스냅샷 및 미지원 실행 파일 거부 테스트
- 수량 내림차순과 동일 수량 이름순 정렬 테스트
- 화면 진입 시 한 번만 자동 호출되는 프론트엔드 테스트
- 새로고침 중 중복 호출 방지와 실패 시 기존 결과 유지 테스트
- 결과 있음, 빈 상태, 각 오류 상태의 렌더링 및 한국어·영어 번역 테스트
- 프로세스 접근 권한이 읽기 전용으로 유지되는 보안 회귀 테스트

자동 테스트 후에는 오프라인 또는 비공개 게임 세션에서 통제된 수량 변경 검증을
수행하고 별도 체크리스트에 실행 파일 해시, 비교 결과 및 판정을 기록한다.

## 완료 조건

- 사이드바에서 `아이템 분석` 화면으로 이동할 수 있다.
- 화면 진입과 수동 새로고침이 검증된 일반 아이템 스냅샷을 읽는다.
- 모든 일반 아이템 중 수량 900 이상인 항목만 정확히 표시한다.
- 오류와 빈 결과가 명확히 구분된다.
- 읽기 전용 안전 규칙과 자동 테스트가 통과한다.
- 실제 게임 2.0.2 통제 검증이 완료되기 전에는 릴리스 기능으로 활성화하거나
  호환성을 주장하지 않는다.
