# Granblue Fantasy: Relink 2.0.2 로컬 진 레이아웃 검증

검증일: 2026-07-18, 외부 리더 재검증 2026-07-20
대상: Windows x64, 오프라인 로컬 플레이
상태: 진 12칸 검증 완료. 무기·가호석·Master Trait·Summon trait은 미검증.

## 버전 고정 정보

- 게임: Granblue Fantasy: Relink Endless Ragnarok 2.0.2
- 실행 파일: `D:\SteamLibrary\steamapps\common\Granblue Fantasy Relink\granblue_fantasy_relink.exe`
- 실행 파일 SHA-256: `63340832BCF731FBC97796F686B05C988418E83D451D4A49B2244A85D00E297F`
- 진단 DLL SHA-256: `31B327EA285E33C73DDAD2156453E997DC5656B3C2669E8A60261651C8D84268`
- 플레이어 갱신 시그니처: `55 41 57 41 56 41 54 56 57 53 48 83 EC 70 48 8D 6C 24 70 48 C7 45 F8 FE FF FF FF 80 B9 BC 5E 00 00 00`
- 실행 파일 내 일치 수: 1 (`0xA2AA00`)

진단 빌드는 `ReadProcessMemory`만 사용해 로컬 레코드의 제한된 범위를 복사했다. 게임 메모리와 저장 데이터는 변경하지 않았다.

## 레코드 관계

갱신 함수의 첫 번째 인자는 로컬 캐릭터 레코드다.

- `record + 0x5E60`: 0x250바이트 장비/신원 스냅샷 포인터
- `record + 0x5EA8`: 장비 캐릭터 키
- 스냅샷 `+ 0x000`: 진 배열
- 스냅샷 `+ 0x1E8`: 캐릭터 이름 VBuffer
- 스냅샷 `+ 0x208`: 플레이어 표시명 VBuffer
- 스냅샷 `+ 0x22C`: 파티 인덱스

장비 캐릭터 키는 커스텀 XXHash32(seed `0x0133ECF3`)로 대문자 캐릭터 코드 `PLxxxx`를 해시한 값이다. 실제 표본은 다음과 일치했다.

| 캐릭터 | 코드 | 장비 키 |
|---|---:|---:|
| 라캄 | `PL0300` | `0x079DF0CC` |
| 오이겐 | `PL0500` | `0xDD7A151E` |
| 나루메아 | `PL1400` | `0xE7053919` |
| 유스테스 | `PL2700` | `0x9B15CFB1` |

## 진 배열

배열은 스냅샷 선두부터 12개 고정 슬롯이며 stride는 `0x24`(36)바이트다. 슬롯 `n`의 주소는 `snapshot + n * 0x24`다.

| 슬롯 내 오프셋 | 크기 | 의미 |
|---:|---:|---|
| `0x00` | 4 | 주 특성 ID |
| `0x04` | 4 | 주 특성 레벨 |
| `0x08` | 4 | 보조 특성 ID |
| `0x0C` | 4 | 보조 특성 레벨 |
| `0x10` | 4 | 진 ID |
| `0x14` | 4 | 장비 캐릭터 키 |
| `0x18` | 4 | 진 레벨 |
| `0x1C` | 4 | 획득 순번 |
| `0x20` | 4 | 알림/상태 값 |

빈 슬롯은 ID 필드에 `0x887AE0B0`, 레벨 필드에 0을 사용한다. 정상 슬롯의 장비 캐릭터 키는 레코드의 `record + 0x5EA8` 값과 같아야 한다.

## 통제 차분

나루메아(`0xE7053919`)의 같은 장비 편성에서 대미지 상한 진 한 개만 제거하고 복구했다.

| 단계 | 스냅샷 SHA-256 앞 16자리 | 대미지 상한 기여 | 게임 UI |
|---|---|---:|---:|
| 원본 | `c53f255f00c4d04b` | `15 + 15 + 11 + 15 + 14` | `70 / 65` |
| 슬롯 7 제거 | `b9d2cc13ef10e617` | `15 + 15 + 11 + 15` | `56 / 65` |
| 복구 | `c53f255f00c4d04b` | `15 + 15 + 11 + 15 + 14` | `70 / 65` |

원본과 제거 표본에서 슬롯 7 외의 진 슬롯은 바뀌지 않았다. 슬롯 7은 다음과 같이 변했다.

- 주 특성 ID: `0xDC584F60` → `0x887AE0B0`
- 주 특성 레벨: `14` → `0`
- 진 ID: `0xEE732781` → `0x887AE0B0`
- 장비 캐릭터 키: `0xE7053919` → `0x887AE0B0`
- 진 레벨: `14` → `0`

같은 56 상태가 두 번 호출됐을 때 스냅샷 해시가 동일했다. 따라서 생산 훅은 캐릭터 키별 최신 이벤트를 캐시해 동일한 스냅샷을 다시 보내지 않는다.

## 거부 조건과 현재 범위

생산 디코더는 다음 입력을 정상 결과로 만들지 않는다.

- 0x1B0바이트보다 짧은 진 배열
- 12개를 초과하는 슬롯 접근
- 정상 슬롯의 장비 캐릭터 키가 레코드 키와 불일치
- 특성 ID가 0 또는 `0x887AE0B0`
- 특성 레벨이 1..=10,000 범위 밖

이번 단계의 `Complete`는 **진 12칸의 주·보조 특성이 완전히 디코딩됨**을 뜻한다. 무기, 가호석, Master Trait, Summon trait은 통제 표본으로 격리되지 않았으므로 추정해서 합계에 넣지 않는다. UI도 이를 전체 장비 합계가 아닌 진 특성 합계로 표시해야 한다.

초기 연결 시 확인된 파티 4명과 장비 화면에서 갱신된 캐릭터는 같은 콜백으로 캡처된다. 전체 보유 캐릭터를 별도 로스터 컨테이너에서 선행 열거하는 경로는 아직 검증되지 않았으므로, 현재 구현은 실제로 갱신 콜백이 관찰된 로컬 캐릭터만 표시한다.

## 외부 읽기 전용 리더 검증

개발 전용 프로브를 `DJEETA_EXTERNAL_READER_PROBE=1`로 활성화하고, 훅이 보낸 스냅샷을 정답값으로 삼아 별도 프로세스의 `ReadProcessMemory` 결과와 비교했다. 외부 리더는 `PROCESS_QUERY_INFORMATION | PROCESS_VM_READ`만 요청하며 게임 메모리에 쓰지 않는다.

- 고정 EXE SHA-256: `63340832BCF731FBC97796F686B05C988418E83D451D4A49B2244A85D00E297F`
- 고유 서명 일치 RVA: `0x2D7E60`
- 로컬 파티 키 전역 RVA: `0x7C23878`
- 플레이어 관리자 전역 RVA: `0x7C24980`
- 파티 슬롯: 4개, 로컬 키 stride `0x10`
- 해시 버킷의 첫 노드와 끝 노드가 같은 단일 노드 버킷도 키를 먼저 비교해야 한다. 끝 노드 여부를 먼저 검사하면 유효한 로컬 캐릭터를 누락한다.

파티 4명 모두에서 현재 스냅샷이 훅과 일치했고, 각 캐릭터의 진 변경 후에도 새 다이제스트가 일치했다. 변경 중 두 번 읽은 값이 다르면 `UnstableRead`로 보류하고 다음 안정된 표본만 비교했다.

| 파티 슬롯 | 장비 키 | 변경 검증 | 결과 |
|---:|---:|---|---|
| 0 | `0x9B15CFB1` | 진 변경 전·후 | `MATCH` |
| 1 | `0xDD7A151E` | `19 / 038bbb999d663830` → `18 / c4f43dc47550de27` | `MATCH` |
| 2 | `0x079DF0CC` | `20 / 0ddd57d455985e31` → `21 / ed6b850a818398fd` | `MATCH` |
| 3 | `0xE7053919` | `18 / 8f2db6f3617552ac` → `19 / 7330810e88da7446` | `MATCH` |

프로세스 재실행은 PID `13812`에서 시작해 `6052`, `24620`, `5340`으로 세 차례 검증했다. 매번 고정 해시와 서명을 다시 확인하고 네 슬롯을 재탐색한 뒤 모두 `MATCH`에 도달했다. 새 훅 파이프 세션이 연결되면 이전 세션의 비교 기준을 폐기하여 로딩 직후 오래된 기준값으로 인한 거짓 불일치를 방지한다.

이 검증은 읽기 전용 외부 리더가 현재 파티의 진 스냅샷을 찾을 수 있음을 입증한다. 훅 제거 또는 전체 장비 소스 지원을 확정하는 검증은 아니다.
