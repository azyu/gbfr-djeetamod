# 전체 캐릭터 진 로스터 조사 기록

## 현재 상태

- 공개 버전은 `0.1.1`이다.
- `main`의 기준 커밋은 `20db5e5 chore: release Djeeta MOD 0.1.1`이다.
- 이번 조사는 `codex/all-character-sigil-roster` 브랜치에서 수행했다.
- 조사 결과를 생산 UI에 연결하지 않았으며, 현재 장비 분석 화면의 동작은 바뀌지 않았다.
- 브랜치는 후속 조사를 위해 삭제하거나 `main`에 병합하지 않고 유지한다.

## 문제

현재 장비 분석은 파티에 들어왔거나 장비 갱신 콜백이 관찰된 로컬 캐릭터만 표시한다. 따라서 실제로 해금되어 사용할 수 있지만 현재 세션에서 콜백이 발생하지 않은 캐릭터의 진 정보는 목록에 나타나지 않는다.

기존 훅의 장비 콜백은 한 캐릭터의 진 12칸을 정확히 읽을 수 있지만, 저장 데이터에 존재하는 모든 해금 캐릭터를 먼저 열거해 주지는 않는다. 이 문제는 `0.1.1`에서 새로 생긴 회귀가 아니라 기존 캡처 범위의 한계다.

## 승인된 목표

가능하면 다음 동작을 구현하는 것이 목표였다.

1. 현재 저장 데이터에서 해금되어 사용 가능한 모든 캐릭터를 자동으로 찾는다.
2. 파티 또는 장비 화면 방문 여부와 무관하게 각 캐릭터의 진 정보를 읽는다.
3. 일시적으로 읽을 수 없는 캐릭터도 로스터에는 남기고 `사용 불가` 상태로 표시한다.
4. 기존 콜백 결과를 우선하고, 외부 읽기 전용 결과를 보조 정보로 병합한다.

단, 게임 메모리나 저장 데이터에 쓰지 않고, 잠긴 캐릭터를 해금 캐릭터로 잘못 표시하지 않는 것이 선행 조건이었다.

## 구현한 검증 프로브

생산 기능을 바로 만들지 않고, 먼저 플레이어 관리자 테이블이 해금 로스터로 사용할 수 있는지 검증하는 디버그 전용 프로브를 구현했다.

- 활성 조건: 디버그 빌드이면서 `DJEETA_EXTERNAL_READER_PROBE=1`
- 게임 대상: Granblue Fantasy: Relink Endless Ragnarok 2.0.2
- 고정 EXE SHA-256: `63340832BCF731FBC97796F686B05C988418E83D451D4A49B2244A85D00E297F`
- 프로세스 권한: `PROCESS_QUERY_INFORMATION | PROCESS_VM_READ`
- 게임 메모리·저장 데이터 쓰기: 없음
- 후보 버킷 상한: 4,096
- 전체 노드 탐색 상한: 1,024
- 순환 목록, 중복 키, 레코드 자체 키 불일치, 잘못된 스냅샷 포인터를 별도로 감지
- 알려진 캐릭터 해시 30개만 장비 캐릭터 후보로 분류
- 50ms 간격으로 두 번 읽어 `stable`, `unavailable`, `unstable`, `invalid` 상태로 분류
- 5초마다 후보 개수와 짧은 SHA-256 다이제스트만 출력

로그에는 로스터 레코드 주소, 원시 진 값, 플레이어 이름, 저장 데이터 내용을 추가하지 않았다. 프로토콜, 훅 메시지, 생산 장비 상태, React UI도 변경하지 않았다.

## 테스트 주도 구현 기록

구현은 실패 테스트를 먼저 확인한 뒤 최소 코드로 통과시키는 순서로 진행했다.

| 커밋 | 내용 |
|---|---|
| `6cf0303` | 전체 캐릭터 진 로스터 설계 |
| `73ffed8` | 생산 구현 전 검증 프로브 계획 |
| `d5a45ae` | 장비 캐릭터 해시와 액터·펫 해시 구분 테스트 |
| `b47cfc3` | 경계가 있는 후보 관리자 탐색기 |
| `2b8e936` | 5초 주기 읽기 전용 로스터 증거 출력 |
| `600120b` | 실제 게임 수동 검증 결과 기록 |

추가된 회귀 테스트는 다음을 검증한다.

- 빈 버킷, 단일 노드, 충돌 체인 탐색
- 4,096개보다 많은 후보 버킷 거부
- 연결 목록 순환과 1,024개 초과 노드 거부
- 중복 캐릭터 키 축약 및 보고
- 유효하지 않은 스냅샷 포인터가 있어도 후보 소속 정보 보존
- 노드 키와 레코드 자체 키가 다르면 결과에서 제외
- 안정·읽기 불가·불안정·디코딩 불가 스냅샷 분류
- 디버그 환경변수 게이트와 읽기 전용 Windows 권한 유지

## 자동 검증 결과

다음 명령이 모두 성공했다.

- `npm.cmd run format-check`
- `npm.cmd run lint`
- `npm.cmd run tsc`
- `npm.cmd test -- --run`: 19개 파일, 74개 테스트 통과
- `npm.cmd run build`
- `cargo build --release --locked --package hook`
- `cargo test --workspace --all-targets --locked`: 장비 코어 7개, 빌드 도구 13개, Tauri 백엔드 109개, 훅 31개, 프로토콜 2개, 호환성 3개 테스트 통과

검증 환경의 Node.js는 `24.16.0`이었다. 프로젝트가 지원하는 표준 도구 체인은 Node.js 20이므로, 이 결과가 지원 버전을 변경하지는 않는다.

## 실제 게임 검증

2026-07-22에 새 게임 프로세스 PID `21020`에서 검증했다.

| 항목 | 결과 |
|---|---:|
| 게임 UI에 표시된 해금 캐릭터 | 24명 |
| 후보 관리자에서 발견된 알려진 캐릭터 | 30개 |
| 기타 자체 일관 키 | 6개 |
| 후보 버킷 | 64개 |
| 중복 키 | 0개 |
| 자체 키 불일치 거절 | 0개 |
| 안정적으로 두 번 읽힌 알려진 스냅샷 | 30개 |
| 읽기 불가·불안정·디코딩 불가 | 0개 |

고정 실행 파일 해시와 고유 탐색 서명은 일치했고 탐색도 상한 안에서 끝났다. 그러나 플레이어 관리자 테이블은 해금된 24명만 담지 않고 알려진 캐릭터 레코드 30개를 모두 담고 있었다. 진이 없는 0개 소스 스냅샷도 정상적으로 디코딩되므로, 빈 장비 여부를 잠금 상태로 해석할 수도 없다.

따라서 수동 검증의 정확한 로스터 일치 조건은 첫 프로세스에서 `FAIL`이었다. 하드 게이트 규칙에 따라 두 번째·세 번째 프로세스 반복과 비파티 캐릭터 진 변경 시험은 실행하지 않았다.

## 결론과 기술적 한계

현재 확인한 플레이어 관리자 테이블만으로는 다음 두 집합을 구분할 수 없다.

- 해금되어 실제로 사용할 수 있는 24명
- 게임이 미리 생성해 둔 알려진 캐릭터 레코드 30개

그러므로 30개 후보를 그대로 생산 UI에 표시하면 잠긴 캐릭터 6개를 잘못 노출한다. 이는 사용자가 요청한 “현재 사용할 수 있는 모든 캐릭터”의 의미와 맞지 않으므로 적용하지 않았다.

현재 안전하게 유지할 수 있는 범위는 기존 콜백 기반 목록이다. 이 방식은 표시되는 캐릭터의 진 정보는 검증됐지만, 세션에서 관찰되지 않은 해금 캐릭터를 빠뜨릴 수 있다.

## 후속 작업 재개 조건

생산 기능을 재개하려면 다음 중 하나가 별도로 검증되어야 한다.

1. 플레이어 관리자 레코드 안의 해금 상태 필드
2. 해금 캐릭터만 포함하는 별도 로스터 컨테이너
3. 저장 데이터의 해금 상태를 안전한 읽기 전용 방식으로 판별하는 경로
4. 게임 UI의 24명과 정확히 대응하는 다른 안정된 런타임 신호

새 신호는 최소 세 개의 새 게임 프로세스에서 동일한 결과를 내야 하며, 잠긴 캐릭터, Id 변신 액터, Ferry 펫, 온라인 플레이어 신원을 별도 캐릭터로 포함하지 않아야 한다. 그 검증이 끝난 뒤에만 5초 생산 리더, `사용 불가` 상태, 콜백 우선 병합, React 로스터 UI를 별도 계획으로 구현한다.
